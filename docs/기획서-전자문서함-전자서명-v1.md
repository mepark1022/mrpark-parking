## 1. 배경 & 목적

**현재 상태 (실측)**
- parking v2: `employees` 마스터 존재(sql/v2/01). 단 **근로계약 회차(세그먼트) 마스터 없음**, 전자문서/서명 코드·기획 전무.
- mrpark-2.0: 근로계약 회차 로직(`contract.ts`)·급여엔진 존재(별도 DB). 서명은 "어드민이 서명완료 체크"하는 상태플래그일 뿐.
- 즉 인사 도메인이 **두 시스템에 분산**되어 있고, 계정·인사 체계 기획서는 이를 **완전 신규 인스턴스로 재신축**(결정 #7)할 계획.

**해결할 문제**
1. 임시크루 일/기간 단위 계약갱신이 수기 → 발송·서명·연동 자동화.
2. 모두싸인 외부 의존(건당 비용) → 전자서명을 **시스템 내부에 내재화**.
3. 계약·인사 문서가 종이/구두로 흩어져 추적·보관·감사 불가.

## 3. 전체 플로우 (거처 무관)

```
[어드민]  문서유형 선택 → 대상 크루 선택 → 템플릿 본문 생성 → 발송 (status=sent)
   │
   ▼
[크루앱 메인]  me.talk 알림: "서명 대기 문서 1건이 있어요" (앱 진입 시 조회; 푸시는 네이티브)
   │
   ▼
[크루앱 설정 > 전자문서함]  본문 확인(viewed) → 동의 체크 → 캔버스 서명 → 제출(signed)
   │
   ▼
[어드민]  서명완료 수신 → 계약 자동연동 (상태·회차 갱신, §7)
   │
   ▼
[대시보드]  미서명 / 서명완료 / 만기 D-7 KPI
```

## 4. 어드민 기능
- 문서 작성·발송 (유형 / 대상 emp / 템플릿 본문 / 계약서는 회차·기간·임금)
- 히스토리: 발송(sent) → 열람(viewed) → 서명완료(signed) → 만료(expired)/회수(canceled) 타임라인
- 계약 자동연동(§7) — 임시크루 갱신 자동화의 핵심
- 대시보드 KPI: 미서명 N / 만기 D-7 크루 N / 월 서명완료율

## 5. 크루앱 기능
- 메인 me.talk 알림 (R6 항상노출 톤). 웹=진입 시 조회, **푸시=네이티브 단계**
- 설정 > 전자문서함 = **연차신청 형제 메뉴**(`/crew/settings` → `/crew/documents`)
- 목록(미서명 상단고정) → 본문 확인(viewed 기록) → ①동의 체크 ②캔버스 서명 ③제출
- 서명완료 문서는 감사로그 포함 PDF로 보관/다시보기

## 6. 전자서명 효력 보강 (감사 메타데이터)

> 자체 캔버스 서명은 그림만 캡처하면 효력 약함. 아래를 함께 박으면 비용 0으로 방어력 확보.
> 전자서명법(2020 개정): 본인 의사·귀속 입증 시 일반 전자서명 효력 인정. **최종 효력은 노무사 1회 확인 권장.**

| 보강 | 내용 |
|------|------|
| 타임스탬프 | 서명 시각(서버 KST) |
| 기기/접속 | User-Agent + IP(마스킹) |
| 문서 해시 | 본문 **SHA-256** → 위변조 검증 |
| 명시 동의 | "본 내용에 동의합니다" 체크 기록 |
| 감사로그 PDF | 서명완료 PDF 하단에 누가/언제/어디서/해시 인쇄 |
| 본인 식별 | emp_no + 로그인 세션(연락처 인증) 귀속 |

## 7. 계약 자동연동 규칙

| 문서 | 서명완료 시 |
|------|------|
| 근로계약서 | 신규 회차 세그먼트 active 전환, 직전 세그먼트 종료. 임시크루=기간 세그먼트 누적 |
| 사직서 | employees status=퇴사, resign_date 동기화, 활성 계약 종료 |
| 시말서 | 직원 문서 이력 보관(계약 상태 변동 없음) |

> 연동 실패 시 서명은 유효 처리 + 연동 에러 로그 + 어드민 알림(서명/연동 분리, 정합성 우선).

## 8. DB 스키마 (거처 인스턴스에 추가; 거처 확정 후 실행)

> Storage 버킷은 반드시 Supabase Storage UI에서 생성(SQL INSERT 무음 실패 패턴). org_id 필수 + RLS(DROP POLICY IF EXISTS 후 재생성).

```sql
-- 전자문서 마스터
documents (
  id uuid PK, org_id uuid NOT NULL,
  emp_no text NOT NULL,                 -- 대상 직원 (employees.emp_no)
  doc_type text NOT NULL,               -- labor_contract | resignation | incident_report
  title text NOT NULL,
  body_html text NOT NULL,
  body_hash text NOT NULL,              -- 발송 시점 SHA-256 고정
  contract_id uuid,                     -- 근로계약 연동 대상(거처에 계약 세그먼트 테이블 존재 시)
  status text NOT NULL DEFAULT 'sent',  -- sent|viewed|signed|expired|canceled
  sent_by uuid NOT NULL, sent_at timestamptz NOT NULL DEFAULT now(),
  viewed_at timestamptz, signed_at timestamptz, expires_at timestamptz,
  pdf_path text,                        -- 서명완료 PDF (Storage)
  created_at timestamptz NOT NULL DEFAULT now()
);

-- 서명 + 감사
document_signatures (
  id uuid PK, org_id uuid NOT NULL,
  document_id uuid NOT NULL REFERENCES documents(id),
  emp_no text NOT NULL,
  signature_png text NOT NULL,          -- base64 서명
  agreed boolean NOT NULL DEFAULT false,
  signed_at timestamptz NOT NULL DEFAULT now(),
  ip_masked text, user_agent text,
  body_hash text NOT NULL,              -- 서명 시점 본문 해시(documents.body_hash와 일치 검증)
  created_at timestamptz NOT NULL DEFAULT now()
);
```

상태 색상(브랜드): sent=네이비 #1428A0 / viewed=골드 #F5B731 / signed=그린 #16A34A / expired=레드 #DC2626 / canceled=그레이 #94A3B8.

## 9. 권한 (4단계)
- MANAGE(super_admin·admin): 작성·발송·회수·재발송·히스토리·대시보드
- SELF(crew 본인 emp_no): 본인 문서 조회·서명. 타인 조회 시도 → 403

## 11. ⚠️ 거처 결정 (= 미결 #7 종속)

전자문서함은 employees/근로계약 마스터 위에 얹힌다. 그 마스터의 거처가 아직 미결:
- 후보: ⓐ 완전 신규 인사 인스턴스(계정·인사 체계 기획서 결정 #7) / ⓑ parking v2 흡수 / ⓒ mrpark-2.0
- **직원 데이터는 parking v2 employees에 이미 존재** → 어디로 가든 이전 부담은 작음.
- 거처 확정 전까지: 본 기능 설계(§3~§10)는 거처 무관하게 유효. **거처 정해지면 §8 스키마를 그 인스턴스에 실행하고 §3 플로우를 구현**.
