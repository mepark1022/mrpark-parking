# P0 로그인·계정 단일화 — 확정 설계 v1

> **확정일:** 2026.06.15 (대표) · **상위 정본:** `docs/미팍통합앱_신규기획서_v3.2.md` §6
> **한 줄:** 직원이 **전화번호 하나**로 미팍티켓·미팍 2.0 양쪽에 로그인한다.

---

## 0. 왜 필요한가 (선결 점검으로 확인된 현실)

직원이 들어갈 시스템이 둘인데(미팍티켓=입출차 / 미팍 2.0=근태·급여), **계정이 시스템마다 따로**다. 실코드 확인 결과 자격증명이 구조적으로 호환되지 않았다:

| 항목 | 미팍 2.0 (정본) | 미팍티켓 parking (비정본) |
|---|---|---|
| 인증 풀 | Supabase `ohonmvug…` | Supabase `xwkatswg…` (**별도 풀**) |
| 로그인 입력 | 전화번호 | 이메일/전화/사번 자동판별 |
| crew 내부이메일 | `{전화}@mepark.internal` | `{사번}@mepark.internal` (**규칙 상이**) |
| 관리자 | 전화+PIN | 실제 이메일+비번 |
| 초기 비번 | 123456 | 전화 뒤4자리+12 |
| 동기화 | **없음** | — |

→ "한 계정 양쪽 진입"은 ①인증 풀 2개 ②내부이메일 규칙 상이 ③초기 비번 상이 를 정합시켜야 성립.

---

## 1. 확정 결정 (대표 2026.06.15)

1. **로그인 ID = 전화번호로 통일.** 양쪽 내부이메일 규칙을 `{전화}@mepark.internal`로 단일화(2.0 기존 규칙에 parking을 맞춤). **parking 사번 기반 규칙 폐기.**
2. **관리자도 전화번호 로그인.** 단 **super_admin(대표) 이메일 로그인은 예비 경로로 유지**(긴급/잠금 대비).
3. **계정 정본 = 미팍 2.0.** 발급·비번변경·퇴사(ban)는 **2.0에서만**. parking은 **종속 복제본(복사)**만 보유 — parking 직접 발급 API는 §3-5에 따라 **신규 실행 동결**.
4. **자격증명 정합 = 단방향 복사(미러 프로비저닝).** 2.0에서 변동 발생 시 parking 인증에 **동일 (전화-이메일·비번·ban 상태)** 를 upsert. parking 로그인 런타임은 **자기 풀에서 그대로 인증**(라이브 무변경 지향).
5. **초기 비밀번호 = 전화 뒤4자리+12로 통일**(§6). 〔추천값 — 변경 시 알려주세요〕
6. **도메인 = 현행 양 주소 병행 유지**(admin.mepark.kr / hr.mepark.kr). 도메인 통합·`crew.mepark.kr` 신설은 **P3(네이티브 전환)로 보류**.

---

## 2. 남는 주의 (사일로·PII)

- **전화번호 유일성 필수.** 전화번호가 곧 로그인 ID이므로, 양쪽 employees에서 전화번호 중복이 있으면 로그인이 충돌한다 → 정본(2.0)에서 유일성 보장.
- **전화번호 변경 = 로그인 ID 변경.** 인증 동기화 별도 트랙(2.0 TODO 기존 항목)과 연동.
- **PII 무관.** 내부이메일은 인증용 합성값이다. "전화번호 저장 금지"는 **방문객 한정** — 직원 인증과 혼동 금지.

---

## 3. 구현 파트 (순서)

- **파트1 — 복사기(백엔드).** 2.0 계정 변동(발급·비번변경·퇴사) → parking 인증에 전화-이메일 기준으로 자동 upsert. **신규 계정부터 양쪽 자동 일치.** (라이브 무변경, 가장 안전 → 먼저)
- **파트2 — parking 로그인 전환 + 기존 계정 이사.** parking 로그인 규칙을 전화 기반으로 전환, 기존 사번-이메일 계정을 전화-이메일로 **1회 마이그레이션**. 관리자 전화 로그인 + 대표 이메일 예비.
- **파트3 — 동결 + 검증.** parking 직접 발급 통로 동결(§3-5), 한 직원이 **같은 전화/비번으로 양쪽 로그인** 되는지 실기기 확인.
