## ⚡ v2 → v3 변경 요지 (왜 갱신했나)

v2는 **"하나의 백엔드"를 전제**로 작성됐으나, 실제 개발은 두 시스템으로 갈라졌다 — 입출차(미팍티켓)는 그대로, 인사·근태·급여·전자문서함은 별도 레포·별도 Supabase(미팍 2.0)로. 그 결과 **DB 2개 + 근태·마감 기능 중복**이라는, v2가 풀려던 "사일로"가 부분 재발한 상태다.

v3는 이를 정면으로 다룬다:

1. **통합의 정의를 재정립** — 통합은 "화면 개수 축소"가 아니라 **백엔드(계정·DB·권한) 단일화**다. 관리자앱·크루앱 2개는 설계대로 유지한다.
2. **현재 구축 현황(as-built)을 정확히 기록** — 무엇이 어느 시스템에 있는지, 무엇이 중복인지.
3. **통합 전략을 C안(점진 통합)으로 확정** — 로그인부터 단일화하고, **네이티브 전환 시점을 'DB 통합 디데이'로 못 박는다.**
4. **문서가 다시 낡지 않도록** 전략(기획서)과 as-built(TODO)의 역할을 분리한다.

## 1. 통합의 정의 (가장 중요 — 오해 제거)

> **"앱이 2개면 통합한 의미가 없는 것 아니냐"** 라는 우려에 대한 공식 답.

**통합 = 백엔드(계정 · DB · 권한)를 하나로 모으는 것.** 화면(프런트 인터페이스)의 개수와는 무관하다.

| | 구버전(사일로) | 통합앱(목표) |
|---|---|---|
| 백엔드 | 시스템 3조각(파트너앱·키오스크·인사 따로) | **단일 API + 단일 DB** |
| 계정 | 시스템마다 별도 | **단일 employees 마스터** |
| 권한 | 제각각 | **4단계 통합 권한 + org_id** |
| 화면 | 여러 개 | 관리자앱 + 크루앱 + 고객페이지 (**그대로 OK**) |

- **관리자앱(본사·PC·회계)** 과 **크루앱(현장·폰·입출차)** 을 분리하는 것은 **올바른 설계**다. 한 화면에 욱여넣으면 오히려 보안·UX가 나빠진다. (Toast·Square 등 운영 솔루션도 매니저앱/직원앱을 분리하되 백엔드는 하나다.)
- 따라서 **"2개 앱"은 문제가 아니다.** 문제는 **"2개 백엔드(DB)"** 이며, v3의 모든 로드맵은 이 하나를 푸는 데 맞춰진다.

## 3. 통합 전략 — C안: 점진 통합 (확정 2026.06.15, 대표)

### 3-1. 결정 요약

세 방향(A 즉시 완전통합 / B 2 DB 영구병행 / C 점진통합) 중 **C 확정.**
근거: 라이브로 도는 입출차를 지금 건드리지 않아 **안전**하고, 어차피 **네이티브 전환 시 API·앱 셸을 재정비**하므로 그때 DB까지 합치는 것이 가장 자연스럽다. 단일 개발자·라이브 운영 상황에 가장 적합.

### 3-2. 단계 (Phase)

| Phase | 내용 | 통합 수준 |
|---|---|---|
| **P0. 로그인·계정 단일화** | 한 계정으로 양쪽 진입(통합 로그인 §6). employees 정본 = 2.0 고정 | ★ 표면 |
| **P1. 정본 확정 + 단방향 참조** | 근태/마감 등 중복 기능의 정본 결정(§3-4). 비정본 쪽은 읽기/이관만, 신규 쓰기 금지 | ★★ |
| **P2. 데이터 브리지** | 입출차 실적(매출·대수)을 통합 회계/대시보드로 단방향 전달(읽기 전용 동기화) | ★★ |
| **P3. 네이티브 전환 = DB 통합 디데이** | 단일 Supabase로 수렴 + 관리자/크루 네이티브 앱. **진짜 단일 시스템 완성** | ★★★ |

### 3-3. ⛳ DB 통합 디데이 — C가 B로 변질되지 않게 하는 단 하나의 못

> **"네이티브 전환 = DB 단일화"를 로드맵에 하드 커밋한다.**
> 이 못이 없으면 "나중에 합치자"가 무한정 미뤄지고, 그 순간 C안은 대표가 우려한 **영구 병행(B)** 으로 굴러떨어진다.
> → 양 레포 TODO 최상단·§9·§10 체크리스트에 동일 문구로 박는다.

### 3-4. 정본 규칙 (사일로 재발 방지)

1. **employees(직원 마스터) 정본 = 미팍 2.0.** 다른 어떤 시스템도 직원 레코드를 **독립 생성/수정하지 않는다.** (전자문서함이 2.0로 이관된 것과 동일 원칙)
2. **단방향 참조.** P3 전까지 parking이 직원 정보가 필요하면 2.0을 **읽기**만 한다(또는 P0 단일 계정으로 식별). 양방향 쓰기 금지.
3. **근태·마감 정본 = 미팍 2.0 (확정, 대표 2026.06.15).**
   - 근태/출퇴근/마감/급여 정본 = 미팍 2.0 (GPS 검증·지각판정·me.talk·급여·전자문서함이 여기 있고, 홈 모드선택 CREW용이 곧 정본 화면).
   - parking의 일보는 **입출차 실적(매출·대수)** 만 책임지고, 그 데이터는 P2 브리지로 통합 회계에 전달. **워커 출퇴근/마감 근태는 2.0으로 수렴, parking 출퇴근·마감은 폐기.**
4. **사업장(store)·방문지·요금 정본 = 미팍 2.0 (확정, 대표 2026.06.15).**
   - employees와 동일 원칙. 사업장에 묶이는 거래처·GPS좌표·반경·근무조·매출마감이 이미 2.0에 있어, store도 2.0이어야 한 `store_id`로 정합한다.
   - ⏳ **과도기 예외(P3 전):** 입출차는 입차 시 **실시간으로 방문지 요금체계를 읽어** 요금을 계산한다. DB가 갈라진 과도기엔 parking이 다른 Supabase store를 실시간 참조할 수 없으므로, **요금·면수·방문지 facet만 parking에 한시 잔류**(입출차 계산용). GPS·거래처·근무조는 2.0. 둘을 **store_id 매핑표**로 연결.
   - 🎯 **P3 병합:** 요금체계·면수·방문지를 2.0 store로 흡수 → 한 사업장 = 한 레코드(요금+면수+GPS+거래처+근무조). 입출차 요금계산도 한 DB 안에서 완결.
5. **중복 쓰기 금지 원칙.** 같은 사실(예: 직원 출근, 사업장 정보)을 두 DB에 동시에 INSERT하지 않는다 — 정본 한 곳만.

### 3-5. ⛔ 재분열 금지 가드레일 (통합을 되돌리지 않기 위한 철칙)

> **원칙: 한 번 2.0로 정한 정본은 다시 갈라지지 않는다.** 과도기 잔류를 "또 분열"로 오해·고착시키지 않기 위한 명문 규칙.

1. **신규 도메인은 무조건 2.0에 만든다.** 앞으로 어떤 신규 기능도 parking(비정본)에 새 테이블·새 마스터를 만들지 않는다. 입출차 계열 확장조차 정본 데이터는 2.0.
2. **과도기 잔류 ≠ 영구 분리.** parking에 한시 남는 것은 **요금·면수·방문지 facet뿐**이고, 이는 **P3 병합 대상으로 못 박힌 일몰(sunset) 데이터**다. 새 사실을 여기에 쌓지 않는다(읽기·계산용으로만).
3. **비정본 측 신규 쓰기 동결.** P1부터 parking의 직원·근태·마감·사업장 마스터에 **신규 INSERT 금지**(이관·읽기만). 운영상 불가피하면 정본(2.0)에 쓰고 parking은 참조.
4. **"편하니까 parking에 잠깐"을 금지.** 단기 편의로 비정본에 데이터를 만드는 순간 C안은 B(영구 병행)로 굴러떨어진다 → 발견 즉시 정본으로 회수.
5. **디데이 없는 잔류 금지.** 과도기 잔류 항목은 반드시 **P3 병합 체크리스트(§10)에 등록**된 것만 허용. 등록되지 않은 분리는 설계 위반.

## 5. 역할·권한 체계 (4단계) — 유지

| 역할 | 코드 | 대상 |
|---|---|---|
| 슈퍼관리자 | `super_admin` | 대표 |
| 관리자 | `admin` | 본사 운영팀 |
| 크루 | `crew` | 현장 크루 |
| 현장직원 | `field_member` | 일일/파트타임 |

**권한 분류:** `SYSTEM`(super만) ⊃ `MANAGE`(+admin) ⊃ `OPERATE`(+crew) ⊃ `SELF`(전원·본인 것만).
**범위:** super/admin = 전 사업장 / crew·field = `store_members` 배정 사업장만.
모든 데이터 접근 차단은 **API 레이어**에서 일원화(화면은 메뉴 표시/숨김만).

## 6. 인증 통합 (= Phase 0 진입점)

> P0의 실체. 양쪽(parking·2.0)에 동일 규칙을 적용해 **한 계정으로 양쪽 진입**.

**입력 판별:** `@`→이메일(super/admin) · `010…`→전화번호 정규화 후 employees 조회 · `MP/MPA…`→사번.
**라우팅:** super/admin → admin.mepark.kr · crew → crew.mepark.kr · field → crew(제한모드).
**초기 비밀번호:** 전화번호 뒤4자리+12 통일. 전 역할 변경 가능. `password_changed=false` 시 변경 권장 배너.
**정규화:** 공백·하이픈 제거, 사번 대문자화, 11자리 검증.
**퇴사자:** 퇴사 워크플로에서 자동 ban → 로그인 차단.

> ⚠️ 현재 parking은 admin.mepark.kr, 2.0은 hr.mepark.kr로 도메인이 다르다. P0에서 **계정(employees·인증) 정본을 2.0으로 고정**하고, parking 로그인이 동일 자격증명을 인식하도록 맞춘다. (도메인 통합/리다이렉트 정책은 P0 착수 시 상세화.)

### 7-1-C. 전자서명 동선 (별도 동선 아님 — 크루앱에 녹는다)

> 전자서명은 독립 흐름이 아니라, **크루가 입출차 보던 그 앱에서 받고 그 자리에서 서명**하는 흐름이다.

```
[OFFICE] 전자문서함 발송 (근로계약·사직·시말 / 급여명세)
   │  서버렌더 단일본문 + 주민번호 가드 + me.talk 검수
   ▼
[me.talk 알림]  →  통합 CREW앱 안에서 수신 (별도 앱·별도 로그인 X)
   ▼
[CREW] 문서 확인 → 캔버스 서명(서명형 3종) / 확인 ack(급여명세)
   │  서명 메타 SHA-256 감사
   ▼
[자동 연동]  근로계약 서명→계약 회차 signed · 사직 서명→status=ended · 급여명세→read_at
```

- 서명형 3종(근로계약·사직·시말) = 자체 캔버스 서명 + 감사메타. 확인형 1종(급여명세) = 확인 + 월별 검색.
- **단일 employees 마스터** 위에서 발송 대상·서명자·계약 회차가 한 직원 레코드로 정합. (DB가 갈라져 있으면 이 자동연동이 깨진다 → 통합의 필요성.)

## 8. 핵심 원칙 (불변 — 어떤 작업에서도 준수)

- **전화번호 정책 (2도메인 분리):**
  - 방문객 PII = **DB 저장 금지.** 알림톡 발송 후 즉시 삭제, 마스킹 로그(010****1234)만.
  - 직원 HR = 저장 필요. **암호화 + UI 마스킹 + MANAGE/SYSTEM 복호 + 감사로그.** ("저장 금지"는 방문객 한정 — 직원과 혼동 금지.)
- **멀티테넌시:** 모든 테이블 `org_id` 필수 + RLS + API 미들웨어 이중 방어. crew 쿼리는 `store_id` 스코핑.
- **주차 면수:** `self_spaces + mechanical_normal + mechanical_suv` (lot.total_spaces 사용 금지).
- **API-first:** 비즈니스 로직은 `/api/v1/*`. 웹/네이티브는 같은 API 호출. Supabase 직접 쿼리는 API 라우트 내부에서만(Realtime 구독만 예외).
- **신규 코드 격리:** 기존 v1 코드 미수정. 신규는 `/api/v1/*`, `src/lib/*`, v2 경로.
- **OCR 로드맵:** 웹=Google Vision(서버) → 네이티브=ML Kit(온디바이스, 무료·오프라인).
- **알림톡:** 미팍티켓 차량당 2회(입차+정산완료, 출차알림 없음) / 월주차 D-7 자동+수동.

## 9. 네이티브 전환 = 통합 완성 (P3)

네이티브 전환은 단순 UI 교체가 아니라 **DB 통합 디데이**다. API-first가 이미 지켜져 API·스키마·로직·권한은 100% 재사용되므로, 이 시점에 **단일 Supabase로 수렴**시키는 것이 가장 비용이 낮다.

| 항목 | 재사용 | 재작성 |
|---|---|---|
| API / DB 스키마 / 비즈니스 로직 / 권한 | ✅ 100% | |
| 웹 UI(React) | | ❌ 네이티브 UI |
| Push / 카메라 OCR | | ❌ 네이티브 Push · ML Kit |

**전환 결과:** 관리자 네이티브앱 + 크루 네이티브앱 (2앱) + 고객 QR 모바일웹 → **단일 백엔드.** 비로소 "통합앱"의 정의(§1)가 완성된다.
