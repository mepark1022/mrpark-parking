## 0. 설계 사상 — 왜 바꾸는가

### 기존 한계

| 문제 | 원인 |
|------|------|
| ID/비밀번호 오류 빈번 | 사번·전화번호·이메일 혼재 + `mp+뒤4자리` → `뒤4자리+12` 규칙 혼란 |
| 지원근무·추가/연장근무 미스 다발 | 근무를 **폼(사전 지정)** 으로 입력 → 본사 계획과 현장 실제가 어긋남 |
| 미스를 사람이 찾아야 함 | 분석 수단 없음 |

### 패러다임 전환

```
사전 지정(plan) 기반  →  사후 마감(fact) 기반 + AI 분석
```

- 근태·급여의 정본 = 폼이 아니라 **개인근무마감 데이터**
- 크루가 실제 근무한 시간·장소·특이사항을 본인이 직접 마감
- AI가 1차 검증하고, 어드민은 **이상 건만** 검토
- AI 리뷰를 **본사(어드민)와 개인(크루앱) 양방향**으로 제공하여 기존 한계 돌파

## 1. 역할 체계 (4단계) — 확정

| 레벨 | 역할 | 코드(안) | 권한 |
|------|------|---------|------|
| L0 | **최고경영자** | `owner` | 슈퍼어드민 등록·수정·삭제 가능. 모든 기능 활용 + 수정 |
| L1 | **슈퍼어드민** | `super_admin` | 슈퍼어드민 변경·등록 **불가**. 그 아래 단계 모두 가능. 본사 대시보드 접근 |
| L2 | **어드민** | `admin` | 크루 초대·변경 가능 등 |
| L3 | **크루** | `crew` | 현장 운영 + 개인마감 (+ 지정 시 매출마감) |

**확정 사항:**
- `field_member` **폐지** — 일일알바·파트타임도 역할은 크루로 통합
- 크루는 **전체를 통칭하는 단일 역할.** 주00일(주5일/주6일/주4일/주3일/주2일/주1일/주말주2일/평일주5일 등) 개념은 **역할도 시스템 코드도 아닌, 어드민 직원현황에서 수기 입력하는 필드**
- 기존 근무형태 분류코드(A~D/E~G 23종) 체계는 신체계에서 강제하지 않음

### 1-1. 임시크루 제도 (신설) — 크루 구분은 "기간 유무"

| 구분 | 기간 | 관리 |
|------|------|------|
| **정규 크루** | 기간 정함 없음 | 일반 운영 |
| **임시크루** (대체근무자·일일알바 포함) | **기간 설정으로 관리** | 최초: 어드민(운영팀)이 기간 설정 |

- 기간 내 권한: 본사 지정 사업장에서 **개인근무기록 마감보고 + 해당 사업장 매출 마감** 가능
- 만료 임박 시 **AI가 "갱신하시겠습니까?" 제안** → 어드민 승인 시 갱신 (자동갱신 알람)
- 만료 시 권한 자동 차단

```
store_members.valid_until = NULL  → 정규 크루
store_members.valid_until = 날짜  → 임시크루 (만료 시 차단 + AI 갱신 제안)
```

## 2. 계정 체계 — 확정

### 2-1. ID / 비밀번호 통일

| 항목 | 정책 |
|------|------|
| ID | **핸드폰번호 11자리** (단일 방식 — 판별 로직 불필요) |
| 비밀번호 | **숫자 6자리** (PIN 방식) |
| 초기 비밀번호 | **`123456`** 통일 |
| 첫 로그인 | **비밀번호 변경 강제** — 변경 완료 전 다른 화면 진입 불가 |
| 변경 안내 | AI 비서가 즉시 변경 안내 (강제 플로우의 안내 역할) |

**변경 사유:** 기존 ID/비번 오류가 빈번하여, 연락처 ID + 숫자 6자리로 변경·통일.

**보안 근거:** 전화번호는 동료 간 공개 정보이므로 초기PW 상태 방치 시 타인 로그인 위험 → 강제 변경으로 차단. 급여명세서·계좌정보가 크루앱에 포함되므로 필수.

### 2-2. 계정 생성 / 초대

- **직원등록 기능**을 통해 권한 부여 → 팀원 초대
- 최고경영자 최초 시드 계정: **010-9535-7333 / `442812`** → 이하 팀원 초대 가능
- 전화번호 ID는 **unique 제약 + 등록 시 중복 검사** 필수 (동일 번호 2명 케이스 실존)
- Supabase 내부 매핑(안): `{phone11}@mepark.internal`

## 3. 마감 체계 (이원화) — 확정

| 구분 | 누가 | 입력 내용 |
|------|------|----------|
| **개인마감** | 팀 전원 (각자) | 본인 근무시간 + 근무처 + 특이사항 |
| **매출마감** | 해당일 **지정된 인원** | 매출 사항 + 특이사항 |

### 3-1. 개인마감 검증 플로우 — 확정

```
크루 개인마감 (fact 입력)
  → AI 1차 검증 (출근버튼 시각·지정사업장과 대조 → 이상치 플래그)
  → 어드민: 이상 건만 검토/승인 (전건 승인 아님)
  → 근태·급여 반영
```

### 3-2. 개인마감의 효과

- 지정 근무시간이 중도에 변해도 유연하게 적용 가능
- 타 사업장 지원 / 추가근무 / 연장근무 이슈 대응 가능
- 평일/주5일 크루: 마감보고 시 **AI "연장근무 하시겠습니까?"** → 연장근무 자동반영

## 4. 출퇴근 + AI 비서 — 확정

- **출퇴근 버튼**으로 해당일 지각 등 이슈 판단
- 출근 처리 시 AI 비서: **"00지점 근무가 맞습니까?"** → 오기입 사전 차단
- **평상시 패턴과 다른 근무시간** 기록 시 → AI 확인 (예: "평소보다 이른/늦은 출근입니다. 맞습니까?")
- **평상시와 다른 업장** 선택 시 → AI 확인
- 패턴 기준: 본인의 누적 출퇴근 기록 (주 근무지 · 통상 시간대)

## 5. 크루앱 기능맵

```
[홈]    주차현황 + AI 리뷰 카드 (해당 기간 분석 리뷰)
[운영]  입출차 / 출차요청 (현재까지 내용 유지에 추가)
[하단]  개인마감 버튼 (신설) → 근무시간·근무장소·특이사항 메모
[출퇴근] 출근/퇴근 버튼 → 지각 판정 + AI 지점·시간 확인
[설정]  급여명세서 / 재직증명서 / 연차내역 및 신청 / 급여계좌 등록·변경신청
```

## 6. 본사 대시보드 (슈퍼어드민 이상)

### 6-1. 기간 체계

- 기본 화면 = **해당월** 데이터
- 표시 옆 기간 전환: **전달 / 앞달 / 분기 / 연간 / 특정기간(직접 지정)**

### 6-2. 표시 항목

- 실시간 주차데이터
- 근로계약서 갱신예정자
- 매출 = **발렛비 + 세금계산서 매출금액**
- 세금계산서 발행 매출
- 매입계산서 및 지출내역 총액
- 각 항목 **버튼 클릭 → 상세내역 + 필터 기능**

### 6-3. 사이드 탭 (hr_system_(13).html 분석본 참고하여 구현)

검증된 기능 이식 대상:
- 계약 세그먼트 타임라인 (수습 1/3/4개월 + 계약 6/9/12개월, 만기 D-day 필터)
- **reportBase 신고기준 체계** (4대보험 / 3.3% / 3.3% 타인 / 신고X / 일용직 / 고용&산재 — 타인신고자 구조 포함)
- 은행 입출금 엑셀 업로드 → 적요 정규화 자동분류
- 급여조건 자동계산 (실근무H → 주휴·연차·연장·공휴 산출)

> ⚠️ hr_system_(13).html은 민감정보 평문 포함 — 외부 공유·배포 금지, 로직만 이식.

## 7. 직원현황 (어드민)

- 표시: **본사 부서별 / 등급별 / 크루** 정보 + 전체 목록
- 검색: **이름 / 연락처 / 사번** (사번은 본사 수기입력)
- 수기 입력 필드: 주00일 근무형태, 사번 등
- **직원별 AI 리뷰:** 주 근무지 + 해당월 리뷰 (근태 · 근무지 · 근무시간 · 연차사용 현황)
- 부속 기능:
  - 급여명세서
  - 재직확인서(재직증명서)
  - **4대보험 득실·상실 데이터베이스**
  - 근로계약서 입력폼 (기존 폼/코드 이식)

## 9. DB 스키마 초안 (신규 인스턴스)

```sql
-- 권한은 4개 고정. 근무형태는 수기 텍스트.
profiles (
  id UUID PK REFERENCES auth.users(id),
  org_id UUID NOT NULL,
  role TEXT NOT NULL DEFAULT 'crew',   -- owner / super_admin / admin / crew
  phone CHAR(11) UNIQUE NOT NULL,      -- 로그인 ID (11자리, 중복 검사 필수)
  employee_id UUID REFERENCES employees(id),
  password_changed BOOLEAN DEFAULT FALSE,  -- 초기PW(123456) 변경 여부 → 미변경 시 화면 진입 차단
  last_login_at TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT NOW()
);

employees (
  id UUID PK,
  org_id UUID NOT NULL,
  name TEXT NOT NULL,
  phone CHAR(11),
  emp_no TEXT,                  -- 본사 수기입력 (검색용)
  department TEXT,              -- 본사 부서
  grade TEXT,                   -- 등급
  work_schedule TEXT,           -- 주00일 — 수기 입력 (시스템 강제 없음)
  hire_date DATE,
  resign_date DATE,
  status TEXT DEFAULT '재직'
);

store_members (
  id UUID PK,
  org_id UUID NOT NULL,
  employee_id UUID NOT NULL,
  store_id UUID NOT NULL,
  valid_from DATE,
  valid_until DATE,             -- NULL=정규 크루 / 날짜=임시크루(만료 차단 + AI 갱신 제안)
  can_close_revenue BOOLEAN DEFAULT FALSE,  -- 매출마감 지정 인원
  is_active BOOLEAN DEFAULT TRUE
);

personal_closings (             -- 개인마감 = fact 정본
  id UUID PK,
  org_id UUID NOT NULL,
  employee_id UUID NOT NULL,
  work_date DATE NOT NULL,
  store_id UUID NOT NULL,       -- 실제 근무처 (본인 입력)
  check_in TIMESTAMPTZ,         -- 출근 버튼
  check_out TIMESTAMPTZ,
  work_minutes INT,
  note TEXT,                    -- 특이사항
  ai_flag TEXT,                 -- NULL=정상 / 'time_anomaly' / 'store_mismatch' 등
  admin_status TEXT DEFAULT 'auto',  -- auto(정상 자동반영) / pending(이상-검토대기) / approved / rejected
  UNIQUE(employee_id, work_date, store_id)
);

revenue_closings (              -- 매출마감
  id UUID PK,
  org_id UUID NOT NULL,
  store_id UUID NOT NULL,
  work_date DATE NOT NULL,
  closed_by UUID NOT NULL,      -- 지정 인원
  revenue_data JSONB,
  note TEXT,
  UNIQUE(store_id, work_date)
);
```

> 멀티테넌시: 전 테이블 `org_id` 필수 + RLS. 전화번호는 직원 마스터(employees/profiles)에 한해 저장(로그인 ID 겸용) — 고객 전화번호 저장 금지 정책과 별개.
